# Internationalisation (i18n)

**Owns:** UI string translations for the dashboard, builder, and landing page. Per-event language for guest invitation pages.

**Key files:**
- `lib/i18n.ts` — react-i18next setup; initialises with resources, default language
- `locales/ms.json` — Malay translations (default)
- `locales/en.json` — English translations
- `stores/landing-store.ts` — persists landing page language choice to localStorage

---

## Setup

`lib/i18n.ts` initialises i18next with both locale files bundled at build time (not lazy-loaded). Setup runs once — import `lib/i18n.ts` once at the app root.

```ts
lng: "ms"          // default language
fallbackLng: "ms"  // if a key is missing, fall back to Malay
defaultNS: "translation"
```

Use `useTranslation()` in any React component:

```ts
const { t } = useTranslation();
t("builder.save_status_saving") // → "Menyimpan..."
```

---

## Two languages, two contexts

### 1. Dashboard / builder UI language
Controlled by `landing-store.language` (`"ms"` | `"en"`). The `LanguageToggle` component in the builder header switches this. Language preference persists to localStorage.

To change the active language at runtime:
```ts
import i18n from "@/lib/i18n";
i18n.changeLanguage("en");
```

### 2. Per-event guest invitation language
Each event has a `language` field (`"ms"` | `"en"`) stored in Convex. This controls the language shown to wedding guests on the public `/{slug}` page — independent of what language the dashboard operator has selected.

When rendering the guest page, set the i18next language to `event.language` before rendering guest components.

---

## Adding a new translation string

1. Add the key to `locales/ms.json` (Malay, required — it's the fallback)
2. Add the same key to `locales/en.json` (English)
3. Use `t("your.key")` in the component

Key naming convention: `{module}.{description}` e.g. `builder.save_status_error`, `rsvp.submit_button`, `guest.wish_placeholder`.

---

## Gotchas

- **Malay is the fallback, not English.** If a key exists in `en.json` but not `ms.json`, the English string will NOT be shown for Malay users — i18next returns an empty string. Always add to `ms.json` first.
- **Translations are bundled, not fetched.** Both locale files are imported in `lib/i18n.ts` — no runtime fetch, no loading state, no CDN needed.
- **SSR language mismatch.** Root `app/layout.tsx` sets `<html lang="ms">`. If a guest event is in English, the `lang` attribute won't update automatically — set it explicitly in the guest page layout if SEO matters.
- **`useTranslation()` requires the i18n instance to be initialised.** Import `lib/i18n.ts` early (in a provider or root layout) before any component calls `useTranslation()`.
